执行计划

1. 什么是执行计划：​

  1. 执行计划描述了SQL引擎为执行SQL语句进行的操作
  2. 分析SQL语句相关的性能问题或仅仅质疑查询优化器的决定时，必须知道执行计划
  3. 执行计划常用于SQL调优

2. 执行计划参数描述：​

  1. operation（操作）：Oracle 执行 SQL 的过程
  2. object（对象）：当前操作对应的对象，如：操作的表
  3. cost（成本、代价值）：Oracle 计算出来的一个数值（代价），用于说明 SQL 执行的代价，月底越好
  4. rows（cardinality、基数）：Oracle 估计当前操作的返回结果集
  5. bytes（字节）：执行 SQL 对应步骤返回的字节数
  6. time（耗时）：Oracle 估计的执行 SQL 对于步骤需要的时间

3. 如何查看执行计划：​

  1. 各大软件的执行计划，但这方法不一定是真实的执行计划，信息也不全面（如：DBeaver -> Ctrl + Shift + E）
  2. 使用 sqlplus 窗口（得需要 管理员权限(dba)，没有权限使用第一种吧）：
1. 统计数据级别设置为全部
alter session set statistics_level = ALL

2.执行业务 SQL
select /*+ monitor */ * from ... where ....

3. 为了样式，设置 linesize （linesize：行高  pagesize：每页的数目）
set linesize 200 pagesize 300

4. 查询真实执行计划
select * from table(dbms_xplan.display_cursor(null, null, 'iostats last'))

4. 执行计划顺序：​

  1. operation 缩进判断，缩进最多的最先执行
  2. operation 缩进相同时，最上面的是最先执行的
            执行过程                                       执行顺序
---------------------------------------------------------------------------
SELECT STATEMENT    8                                       八
    NESTED LOOPS    7                                       七
        VIEW            5                                   五
            SORT (AGGREGATE)    3                           三
                TABLE ACCESS (FULL)     1                   一
        VIEW            6                                   六
            SORT (GROUP BY)     4                           四
                TABLE ACCESS (FULL)     2                   二

5. 访问数据的方式：​

1. 全表扫描（TABLE ACCESS FULL）：​

  1. 全表扫描时从第一个区(EXTENT)的第一个块(BLOCK)开始扫描，一直扫描的到表的高水位线(High Water Mark)，这个范围内的数据块都会扫描到

  2. 全表扫描是采用多数据块一起扫的，并不是一个个数据块扫的

  3. 全表扫描慢是针对数据量很多的情况，数据量少的话，全表扫描并不慢的，不过随着数据量越多，需要扫描的数据库越多，自然扫描所需要的IO越多，时间也越多

2. ROWID扫描（TABLE ACCESS BY ROWID）：​

  1. 通过 ROWID 所在的数据行记录去定位数据

3. 索引扫描：​

1. 索引唯一扫描（INDEX UNIQUE SCAN）：​
  1. 索引唯一性扫描(INDEX UNIQUE SCAN)是针对唯一性索引(UNIQUE INDEX)（唯一性约束建立的索引）来说的
  2. 其结果集只会返回一条记录
2. 索引全扫描（INDEX FULL SCAN）：​
  1. 索引全扫描是指扫描目标索引所有叶子块的索引行，但不意思着需要扫描所有的分支块，索引全扫描时只需要访问必要的分支块，然后定位到位于该索引最左边的叶子块的第一行索引行，就可以利用改索引叶子块之间的双向指针链表
  2. 从左往右依次顺序扫描所有的叶子块的索引行
3. 索引范围扫描（INDEX RANGE SCAN）：​
  1. 适用于所有类型的B树索引，一般不包括唯一性索引，因为唯一性索引走索引唯一性扫描
  2. where条件里有一个索引键值列没限定为非空的，那就可以走索引范围扫描，如果改索引列是非空的，那就走索引全扫描
4. 索引快速全扫描（INDEX FAST FULL SCAN）：​
  1. 索引快速全扫描和索引全扫描的相同点：
    1. 扫描所有叶子块的索引行
  2. 索引快速全扫描和索引全扫描区别：
    1. 索引快速全扫描只适应于CBO(基于成本的优化器)
    2. 索引快速全扫描可以使用多块读，也可以并行执行
    3. 索引全扫描会按照叶子块排序返回，而索引快速全扫描则是按照索引段内存储块顺序返回
    4. 索引快速全扫描的执行结果不一定是有序的，而索引全扫描的执行结果是有序的
5. 索引跳跃式扫描（INDEX SKIP SCAN）：​
  1. 可以使那些在where条件中没有目标索引的前导列指定查询条件但是有索引的非前导列指定查询条件的目标SQL依然可以使用跳跃索引
  2. Oracle 承认索引跳跃式扫描没有直接索引查询速度快，但可以这样说，相比于整个表扫描(table scan)，索引跳跃式扫描的速度要快得多

6. 表连接方式：​

1. 排序合并连接（merge sort join）：​

是先将关联表的关联列各自做排序，然后从各自的排序表中抽取数据，到另一个排序表中做匹配

2. 嵌套循环连接（Nested loop join）：​

Nested loops 工作方式是循环从一张表中读取数据(驱动表outer table)，然后访问另一张表（被查找表 inner table，通常有索引）。驱动表中的每一行与inner表中的相应记录join。类似一个嵌套的循环。对于被连接的数据子集较小的情况，nested loop连接是个较好的选择

3. 哈希连接（Hash join）：​

散列连接是CBO 做大数据集连接时的常用方式，优化器使用两个表中较小的表（或数据源）利用连接键在内存中建立散列表，然后扫描较大的表并探测散列表，找出与散列表匹配的行。

4. 笛卡尔连接（Cross join）:​

如果两个表做表连接而没有连接条件，而会产生笛卡尔积，在实际工作中应该尽可能避免笛卡尔积
