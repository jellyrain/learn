表的优化

1. 小表 Join 大表（MapJOIN）：​

将 key 相对分散，并且数据量小的表放在 join 的左边，可以使用 map join 让小的维度表先进内存。在 map 端完成 join。

新版的 hive 已经对小表 JOIN 大表和大表 JOIN 小表进行了优化。小表放在左边和右边已经没有区别

1. 开启 MapJoin 参数设置：​

（1）设置自动选择 Mapjoin

set hive.auto.convert.join = true; -- 默认为true

（2）大表小表的阈值设置（默认 25M 以下认为是小表）：

set hive.mapjoin.smalltable.filesize = 25000000;

2. 大表 Join 大表：​

有时 join 超时是因为某些 key 对应的数据太多，而相同 key 对应的数据都会发送到相同的 reducer 上，从而导致内存不够

1. 空 key 过滤：​

join 前 删除 null 行

条件：

  1. 必须要使用 outer join
  2. null 可以 删除

2. 空 key 转换：​

将 null 值 随机赋值 使得走不同 reducer 中，防止数据倾斜

条件：

  1. 必须要使用 outer join
  2. 某个 key 为空对应的数据很多
  3. null 值 hash 后 不能匹配 正常值

3. SMB(Sort Merge Bucket join)：​

使用 分桶表 来解决 2 的问题

条件：

  1. 最好不要自己设置 reducer 使用默认的 -1，让系统走多少个 reducer

3. Group By：​

默认情况下，Map 阶段同一 Key 数据分发给一个 reduce，当一个 key 数据过大时就倾斜了

并不是所有的聚合操作都需要在 Reduce 端完成，很多聚合操作都可以先在 Map 端进行部分聚合，最后在 Reduce 端得出最终结果

1. 开启 Map 端聚合参数设置：​

（1）是否在 Map 端进行聚合，默认为 True

set hive.map.aggr = true

（2）在 Map 端进行聚合操作的条目数目

set hive.groupby.mapaggr.checkinterval = 100000

（3）有数据倾斜的时候进行负载均衡（默认是 false）

set hive.groupby.skewindata = true

当选项设定为 true，生成的查询计划会有两个 MR Job

第一个 MR Job 中，Map 的输出结果会随机分布到 Reduce 中，每个 Reduce 做部分聚合操作，并输出结果，这样处理的结果是相同的 Group By Key 有可能被分发到不同的 Reduce 中，从而达到负载均衡的目的

第二个 MR Job 再根据预处理的数据结果按照 Group By Key 分布到 Reduce 中（这个过程可以保证相同的 Group By Key 被分布到同一个 Reduce 中），最后完成最终的聚合操作

4. Count(Distinct) 去重统计：​

数据量小的时候无所谓

数据量大的情况下，由于 COUNT DISTINCT 操作需要用一个 Reduce Task 来完成，这一个 Reduce 需要处理的数据量太大，就会导致整个 Job 很难完成

一般 COUNT DISTINCT 使用先 GROUP BY 再 COUNT 的方式替换,但是需要注意 group by 造成的数据倾斜问题

虽然会多用一个 Job 来完成，但在数据量大的情况下，这个绝对是值得的

5. 笛卡尔积：​

尽量避免笛卡尔积，join 的时候不加 on 条件，或者无效的 on 条件，Hive 只能使用 1 个 reducer 来完成笛卡尔积

6. 列处理：​

在 SELECT 中，只拿需要的列，如果有分区，尽量使用分区过滤，少用 SELECT *

7. 行处理：​

1. 谓词下推：​

先过滤 再 join 表

  1. 表达式含有运算（ 运算会新建列 且不是两表字段 ）、表达式含有不确定函数（随机数函数、获取当前时间函数等）不会触发 谓词下推
  2. inner join 不满足 1 的部分 触发 谓词下推
  3. full join 不会触发 谓词下推
  4. left join：
    1. on（ 不满足 1 的部分 ）：
      1. 表达式中 左表字段 不会触发 谓词下推
      2. 表达式中 右表字段 会触发 谓词下推
    2. where（ 不满足 1 的部分 ）：
      1. 表达式中 左表字段 会触发 谓词下推
      2. 表达式中 右表字段 不会触发 谓词下推
  5. right join：
    1. on（ 不满足 1 的部分 ）：
      1. 表达式中 左表字段 会触发 谓词下推
      2. 表达式中 右表字段 不会触发 谓词下推
    2. where（ 不满足 1 的部分 ）：
      1. 表达式中 左表字段 不会触发 谓词下推
      2. 表达式中 右表字段 会触发 谓词下推

2. 子查询：​

谓词查询的缺点：

  1. full join 不走 谓词下推
  2. 谓词下推 只能优化行 不能优化列（只能筛选行，不能选择进一步压缩列，很多时候 关联表 用不着全部列）
  3. 也许会抽风？

所以：通过子查询后，再关联表

缺点：处理不好 可能会出现 多个 job ？
